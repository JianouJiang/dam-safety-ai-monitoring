## FOUR PILLARS EVALUATION

### NOVELTY (20%) -- Score: 4/10

The three-layer hybrid framework (physics + data + knowledge + Bayesian fusion) is a reasonable novelty claim. However, the current implementation undermines every novel element:

- The "GAT" is actually a GCN (see CRITICAL #1 below)
- The "Dempster-Shafer fusion" is actually a weighted average (see CRITICAL #3)
- The "PINN encoding Darcy's law" does not encode Darcy's law (see HIGH #1)

If the paper claims novelty from components that are not what they claim to be, the novelty collapses. A weighted average of a GCN and a monotonicity-constrained MLP is not novel in 2026.

### PHYSICS/SCIENCE DEPTH (40%) -- Score: 2/10

This is the weakest pillar. The plan promises physics-informed learning with governing equations (Darcy seepage, thermal-structural coupling, poro-elastic consolidation). The code delivers:

- A monotonicity constraint (dh/dH > 0) mislabelled as Darcy's law
- No spatial variables in the PINN, so no spatial PDE can be solved
- No temperature physics loss
- Synthetic data generated by a simplified HST model, not physics simulation

The "P" in PINN is nearly absent. A reviewer of Engineering Structures (IF ~5.5) will immediately recognise that `dh/d(water_level) > 0` is not a physics constraint worth publishing -- it is a trivial monotonicity prior. The plan's own words: "the physical governing equations of a dam are not just modelling tools -- they are filters." The current code does not implement these filters.

### CONTRIBUTION (30%) -- Score: 3/10

Impossible to evaluate quantitatively because the experiments have not run to completion. The results directory contains only synthetic data and two model checkpoints. No `metrics_table.csv`, no `full_results.json`, no `score_timeseries.csv`. No comparison against published baselines on real data.

The experimental design has a structural problem: the synthetic data is generated by an HST model (`generate_dam_data.py:118-152`), and one of the baselines is the HST model (`baselines.py:38-84`). This means the HST baseline is being evaluated on data drawn from its own model family. Any comparison showing the proposed method beats HST on this data is methodologically suspect -- the HST baseline has a structural advantage on its own data, and if even it loses, the comparison proves nothing because the data is synthetic.

### RELEVANCY (10%) -- Score: 7/10

The topic (dam safety SHM with AI) is squarely within Engineering Structures' scope. The framing in plan.md is well-targeted. The reference list is appropriate. This is the one pillar that is acceptable.

## CRITICAL ISSUES (Must fix before proceeding)

### CRITICAL #1: "GAT" is actually a GCN -- Misrepresentation

**Files:** `codes/models/gat_lstm.py:20-36`, `codes/models/gat_lstm.py:111-113`

The plan (plan.md, line 101) explicitly promises: *"graph-attention network (GAT) with 2 attention heads."*

The code implements a standard **Graph Convolutional Network (GCN)** using the Kipf & Welling (2017) spectral convolution:

```python
# gat_lstm.py:21 - This is GCN, not GAT
class GraphConvLayer(nn.Module):
    """Graph Convolution Layer: H' = sigma(D^{-1/2} A D^{-1/2} H W)"""
```

There are **zero attention heads**, **zero learnable attention coefficients**, and **zero masked attention** as defined in Velickovic et al. (2018). The `compute_attention_weights` function (line 95-108) computes post-hoc Pearson correlations for visualization -- this is not learned attention.

The class is even named `GATLSTMAnomalyDetector` with the comment "Named GATLSTMAnomalyDetector for API compatibility with experiment runner" (line 113). This is a shortcut dressed up with a misleading name.

**Required action:** Implement an actual GAT layer with learnable attention coefficients per Velickovic et al. (2018), or rename everything to GCN-LSTM and update the plan and novelty claims accordingly. If the latter, the paper loses a significant novelty element (interpretable attention weights showing spatial anomaly propagation).

### CRITICAL #2: 100% Synthetic Data -- Circular Validation Risk

**File:** `codes/data_processing/generate_dam_data.py`

The entire dataset is synthetic. The data generator uses a simplified HST model with hardcoded parameters for a "185m dam" (Three Gorges scale). Anomalies are injected at known locations with known signatures.

Problems:

1. **Circular validation:** The HST baseline (`baselines.py:38-84`) fits the same HST model to data generated by a HST model. Any performance comparison is contaminated.

2. **Known anomaly injection:** The anomaly config is hardcoded (`generate_dam_data.py:382-409`) with known start days, durations, magnitudes, and types. If any model parameter was tuned with knowledge of these anomalies, the results are invalid.

3. **No sensor noise realism:** Real dam sensors have drift, calibration shifts, missing data, and non-Gaussian noise. The synthetic data uses only `np.random.normal` noise.

4. **plan.md contract violation:** The Computational Experiment Contract specifies "Three Gorges Dam monitoring database (proprietary)" as primary and "Jinping-I arch dam or Xiaolangdi dam datasets" as fallback. Neither is used.

**Required action:** Either (a) obtain real dam monitoring data as planned, or (b) use a published benchmark dataset (ICOLD, or datasets from Mata 2011, Salazar 2017), or at minimum (c) acknowledge synthetic data as a proof-of-concept and structure experiments to avoid the circular validation problem (e.g., use a different data generation model than the HST baseline).

### CRITICAL #3: Bayesian Fusion is Not Dempster-Shafer

**File:** `codes/models/bayesian_fusion.py:46-73`

The plan promises "Bayesian evidence combination (Dempster-Shafer or calibrated probabilistic fusion)." The implementation is:

```python
# bayesian_fusion.py:55-63
combined = (w["physics"] * p_norm + w["data"] * d_norm + w["knowledge"] * k_norm)
if sources_high >= 2:
    combined = min(1.0, combined * 1.3)  # hardcoded boost
```

This is a **weighted average with a hardcoded 1.3x multiplier**. Dempster-Shafer theory requires:
- Mass functions (basic probability assignments) for each evidence source
- Belief and plausibility functions
- Dempster's rule of combination with conflict normalization

None of these are implemented. The 1.3x "conflict bonus" (line 63) is an ad-hoc heuristic that has no theoretical basis.

**Required action:** Implement actual Dempster-Shafer combination, or at minimum a principled Bayesian fusion (e.g., calibrated probability combination with proper likelihood models). The current implementation cannot be published under the name "Dempster-Shafer" -- it would be caught immediately by any reviewer familiar with evidence theory.

## HIGH-PRIORITY ISSUES

### HIGH #1: PINN Does Not Encode Governing Equations

**File:** `codes/models/pinn_seepage.py:62-96`

The PINN claims to encode three governing equations (docstring, lines 10-13):
1. "Steady-state seepage: nabla dot (k nabla h) = 0"
2. "Thermal displacement: delta = alpha E DeltaT"
3. "Hydrostatic displacement: delta = f(H)"

What is actually encoded:
1. Monotonicity constraint: dh/d(water_level) > 0 and smoothness: (d2h/d(water_level)2)^2 small
2. Monotonicity constraint: d(displacement)/d(water_level) > 0
3. Nothing for temperature

The PINN takes only 2 inputs: `(water_level, air_temperature)`. There are **no spatial coordinates** (x, y, z), so the Laplacian nabla^2 h that defines Darcy's seepage equation **cannot be computed**. This is not a PINN for seepage -- it is a constrained regression model.

**Required action:** Either (a) introduce spatial coordinates and implement the actual seepage PDE residual, or (b) rename the approach honestly as "physics-constrained neural network" with clearly stated soft constraints. Option (a) is strongly preferred for Engineering Structures.

## MEDIUM-PRIORITY ISSUES

### MEDIUM #1: Knowledge Score Arbitrary Scaling

**File:** `codes/models/run_experiments.py:255`

```python
knowledge_scores_scaled = knowledge_test_scores * 5.0
```

The knowledge layer scores are multiplied by 5.0 to "match other methods' range." This is an arbitrary scaling factor that introduces a hidden tuning parameter. If changed to 3.0 or 8.0, the fusion results and ablation studies would change.

**Required action:** Use a principled calibration method (e.g., Platt scaling, isotonic regression) to map knowledge scores to the same probability space as other layers.

### MEDIUM #2: Inconsistent Sequence Lengths in Baselines

**Files:** `codes/models/baselines.py:132` vs `codes/models/gat_lstm.py:116`

- GCN-LSTM uses `seq_len=14`
- LSTM Autoencoder baseline uses `seq_len=30` (hardcoded default)
- The experiment runner (`run_experiments.py:340`) passes `seq_len=14` to LSTM-AE

The baselines.py default and the runner override disagree. More importantly, if the LSTM-AE default were used, the comparison would be on different effective time windows, making it unfair.

**Required action:** Verify the LSTM-AE is actually using seq_len=14 as passed by the runner. Add explicit assertions or logging to confirm.

### MEDIUM #3: Isolation Forest Uses All CPU Cores

**File:** `codes/models/baselines.py:96`

```python
n_jobs=-1  # uses ALL cores
```

Per CLAUDE.md hardware guidelines: "NEVER use all 10 cores -- always reserve 2 for the OS and agent processes."

**Required action:** Change to `n_jobs=8` or `n_jobs=-3`.

### MEDIUM #4: Empty Manuscript

**File:** `manuscript/main.tex`

The manuscript contains only section headers and a placeholder abstract comment. No text has been written. While this is not an error (the Worker hasn't started writing yet), it means no cross-verification between claims and code is possible at the manuscript level.

**Required action:** Begin writing methodology section so claims can be verified against code.
